# サーバー基本設計書

## 概要

　本書は、自動倉庫管理システムにおけるサーバー機能の外部仕様を定義するものである。   
　本システムは、スマートフォン端末からの入出庫依頼および自動倉庫設備からの作業要求・進捗報告を受け付け、JOBの管理および在庫管理を行う。サーバーはスマートフォン端末および自動倉庫設備との通信をHTTP REST APIにより行う。  
　APIの詳細仕様（URI、リクエスト・レスポンス形式、ステータスコード等）は別紙「API仕様書」に定義する。

## システム構成

　本システムは、スマートフォン端末、サーバー、自動倉庫設備で構成する。  
サーバーは、スマートフォン端末および自動倉庫設備からの要求を受け付け、JOB管理、在庫管理、棚管理を行う中心機能として動作する。

```text
スマートフォン端末
    ↓ ↑
  HTTP REST API
    ↓ ↑
サーバー
    ↓ ↑
  HTTP REST API
    ↓ ↑
自動倉庫設備
```

### スマートフォン端末

|項目|内容|
|---|---|
|接続方式|HTTP REST API|
|役割|入出庫依頼、JOB状況確認、商品受取報告|
|主な送信情報|出庫依頼、入庫依頼、商品受取報告|
|主な受信情報|JOB情報、JOB状態、エラー情報|

### 自動倉庫設備

|項目|内容|
|---|---|
|接続方式|HTTP REST API|
|役割|JOB取得、作業実行、進捗報告、エラー報告|
|主な送信情報|オンライン通知、進捗報告、完了報告、エラー情報、入庫要求|
|主な受信情報|出庫JOB、入庫JOB、棚使用状況|

## 業務フロー

### 出庫

1. スマートフォン端末が出庫依頼を登録する。
2. サーバーは出庫JOBを生成する。
3. サーバーは出庫JOBに、商品と自動倉庫を割り当てる。
4. 自動倉庫設備へ出庫JOBを配信する。
5. 自動倉庫設備は出庫作業を開始する。
6. 自動倉庫設備は作業進捗をサーバーへ報告する。
7. 自動倉庫設備は商品を取出口へ搬送する。
8. スマートフォン端末に搬送完了が通知される。
9. 作業者が自動倉庫設備から商品を取り出す。
10. 自動倉庫設備から取出し完了が報告される。
11. サーバーはJOBを完了とする。

```text
スマホ
  ↓ 出庫依頼
サーバー
  ↓ JOB生成
　↓ JOB割当
自動倉庫
  ↓ 作業開始
  ↓ 作業完了
商品搬送
  ↓
商品取出し
  ↓ 取出し
JOB完了
```

### 入庫

1. 自動倉庫設備が入庫対象品種を通知する。
2. サーバーは入庫JOBを生成する。
3. 自動倉庫設備は入庫作業を開始する。
4. 自動倉庫設備は商品を棚へ格納する。
5. 自動倉庫設備は作業完了を報告する。
6. サーバーは在庫情報を更新し、JOBを完了とする。

```text
自動倉庫
  ↓ 入庫要求
サーバー
  ↓ JOB生成
自動倉庫
  ↓ 作業開始
  ↓ 作業完了
商品搬送
  ↓
商品取出し
  ↓ 取出し
JOB完了
```

### エラー発生時

1. 自動倉庫設備でエラーが発生する。
2. 自動倉庫設備はサーバーへエラー情報を通知する。
3. サーバーはエラーログを記録する。
4. JOBの再割当を実行しリトライする。再割当失敗した場合は異常終了する。

``` text
自動倉庫
  ↓ エラー通知
サーバー
  ↓ ログ記録
  ↓　　　　　　　↓
自動倉庫切替　JOB異常終了
```

## 状態の定義（サーバー内部での管理）

### JOB状態

|名称|意味|タイムアウト|
|:---|:---|:---:|
|Unassigned|JOBが作成されたが、商品が割当られていない|-|
|Assigned|JOBに商品が割当られた|-|
|Delivered|自動倉庫にJOBが配信された|○|
|Picking|自動倉庫が出庫を始めた|○|
|WaitOut|出庫が完了したが、商品が取り出されていない|○|
|Putaway|自動倉庫が入庫を始めた|○|
|Completed|JOBが完了した|-|
|Canceled|JOBがキャンセルされた|-|
|Interrupted|JOB実行中の異常により、処理を中断している|-|
|Pending|商品の再割当を待っている|-|
|Aborted|再割当可能な商品が存在せず、JOBが成立しなくなった|-|

### 装置状態

|名称|意味|
|:---|:---|
|Offline|装置の生存が確認できない|
|Available|装置がJOB受付可能|
|Busy|装置がJOBを実行している|

### 在庫状態

|名称|意味|
|:---|:---|
|Stored|商品が棚に保管されている|
|Reserved|商品がJOBに割り当てられて、作業開始を待っている|

## JOB配信仕様

- 出庫JOBは、スマホで生成されたあと、自動倉庫設備へ配信される。
- 出庫JOBの配信方式はプッシュ配信およびプル配信の両方に対応する。
- 入庫JOBは、自動倉庫設備の入庫要求より生成し、改めて自動倉庫設備に対して配信を行う。
- 出庫JOBは、JOB進捗で異常が発生するごとに、JOBの再割当を行う。

### 出庫JOBの商品割当ロジック

　JOB配信ロジックを実行する際に、在庫情報から商品の割当を実行する。
 
1. 当該JOBで指定された品種について、同一品種の商品を、オフラインではない自動倉庫の在庫情報から検索する。
    - 在庫があり、その商品を在庫する自動倉庫設備が作業可能であれば、割り当てを行う。
    - 在庫はあるが、その商品を在庫する自動倉庫設備が、現在作業中であれば未割当状態にする。
    - 在庫がなければ、その出庫JOBを異常終了させる。
  
### JOBの商品再割当

　JOBが異常中断状態になったとき、商品の再割当を実行する。 

1. 異常中断と同時に、割当済みの商品を保管中に戻し、自動倉庫をオフライン扱いにする。
2. 上記割当ロジックと同様に、在庫検索を行う。
    - 在庫があり、その商品を在庫する自動倉庫設備が作業可能であれば、割り当てを行う。
    - 在庫はあるが、その商品を在庫する自動倉庫設備が、現在作業中であれば再割当待ち状態にする。
    - 在庫がなければ、その出庫JOBを異常終了させる。

### 出庫JOB：プッシュ配信

1. サーバーが、割当済みのJOBを配信する際、対象の自動倉庫の状況を再度確認する。
2. 実行可能なままである場合は、配信を実行する。
3. 実行可能でない場合（オフライン等）は、再割当ロジックを実行する。

### 出庫JOB：プル配信

1. 自動倉庫設備は、サーバーへ次の出庫JOBを要求できる。
2. その自動倉庫設備に割り当てられた、未実行のJOBが存在する場合、そのJOBを返却する。
3. 割り当てられたJOBが存在しない場合、その自動倉庫設備が実施可能な、JOBを検索して配信する。
4. その自動倉庫設備が実行可能なJOBが存在しない場合は、JOBなしとして応答する。

### プッシュ配信とプル配信の整合

1. JOBの状態読み取り～更新までを1トランザクションとして考え、先行した配信ロジックが優先してJOB操作権を得る。
2. JOB操作を行った配信ロジックは、JOBの状態を更新する。
3. あとからJOB操作を試みた配信ロジックは、JOBの状態が配信済みであることを確認し、処理のスキップを行う。
    - プッシュ型：処理をスキップし、プッシュを行わない
    - プル型：JOBなしとして応答する

- 異常中断 → 商品再割当後のJOB配信は、プッシュで行う。
- 再割当待ちJOBは、未割当と同じ扱いとする。

### 入庫JOB

1. 自動倉庫設備は、入庫要求イベントをサーバーに通知する。
2. サーバーは要求内容をもとに入庫JOBを登録し、作業対象情報を返却する。

### 出庫JOBと入庫JOB

- 各自動倉庫は、出庫JOBと入庫JOBは独立して実行できる。
- 出庫JOBの割当ロジックに、入庫JOBが混ざらないようにすること。

### JOB配信順序

- 同一設備に複数の未完了JOBが存在する場合、JOBは登録順に処理する。
- 未割当JOBは、登録順に割当を行う。
- 再割当待ちJOBが存在する場合、未割当JOBと同様のロジックで割当を行う。

### 二重配信防止

- 同一JOBは、同一設備に対して重複して処理されないものとする。
- サーバーはJOB状態を管理し、既に処理開始済みのJOBを再配信しない。
- サーバーはJOB作業中の自動倉庫設備に対し、JOBを配信しない。
- JOBに割当済みの商品は、別のJOBに割当ができないようにする。

### 応答の管理

- 【作業開始TO】サーバーは自動倉庫にJOBを配信してから、作業開始報告を受けるまでの時間を管理する
- 【作業完了TO】サーバーは自動倉庫から作業開始報告を受けてから、作業完了報告を受けるまでの時間を管理する
- 【取出し完了TO】サーバーは出庫JOBについて、自動倉庫から作業完了報告を受けてから、取出し完了報告を受けるまでの時間を管理する
- 上記のタイミング中に規定時間を超えた場合はタイムアウト扱いとし、以後タイムアウト処理フローを実行する

### タイムアウト処理

- 各種タイムアウトが発生したとき、サーバーはそのJOBに紐づいた自動倉庫をオフラインとして扱う。
- 作業開始TOがおきたとき、サーバーはそのJOBの品種と同じ商品を、別の自動倉庫から検索する
- 作業完了TO、取出し完了TOについても同様の処理とする
  - ただし、その商品は自動倉庫側で在庫から除去されている前提となるため、次オンライン時はロストする
  - ロストした商品は、運用で対応するものとする（再入庫、管理外搬送など）

### オフライン時の扱い

- 設備がオフライン状態の場合、サーバーは新規JOBを配信しない。
- 設備から異常報告や、タイムアウトがない状態で、オンライン要求を受信した場合、異常中断扱いとする。
- オンライン要求受信時は、在庫状態を更新してから、その自動倉庫が対応可能なJOBの割当を行い、割当後のJOB返却する
- サーバーがオフラインと判断した機器から、オンライン要求以外の通信が来た場合は、すべて通信を拒否する。（未登録端末扱い）

### 異常終了となる条件

- JOBが登録されてから完了されるまでの間に、在庫情報がなくなったとき

### 異常中断となる条件

- JOB割当済みの装置がエラー報告を行った（オフライン化）
- タイムアウトが発生（オフライン扱い）
- オンライン要求がされた（オンライン再要求）
  - これは、タイムアウト前に、エラー報告なく装置が再起動したとみなし、JOBは中断すべきである

## エラー方針・制約

- サーバーは、要求内容に問題がある場合、処理を実行せずエラーとして応答する。
- サーバー内部で異常が発生した場合、処理を中断しエラーとして応答する。
- エラー発生時は、原因調査のためログを出力する。
- 業務エラー発生時は、要求された処理を実行しない。必要に応じてJOBの再割当を試行する。
- タイムアウト発生時は、相手装置をオフライン状態へ遷移させ、JOBの再割当を試行する。
- システムエラー発生時は、相手装置をオフライン状態へ遷移させ、担当JOBを異常終了とする。

### 画面：業務エラーの定義

- 出庫：存在しない商品が指定された
- 出庫：在庫が不足している
- 状態遷移条件を満たしていない

### API：業務エラー

- 状態遷移条件を満たしていない
- 入庫：存在しない商品が指定された
- 入庫：棚容量を超過する場合

### タイムアウトの定義

- レスポンスが一定時間返却されない
- 時限付き状態が設定時間を超過した

### システムエラーの定義

- API契約違反（400）
- 認証設定異常（403）
- JOB管理・指定の不正（404）
- 状態同期異常（409）
- その他内部エラー（500）

### エラーログの出力

エラー発生時は以下の情報をログへ記録する。

- 発生日時
- 要求元設備
- 対象JOB
- エラー内容
