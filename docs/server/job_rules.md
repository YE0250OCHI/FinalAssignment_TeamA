# JOB設計

## 概要

- JOBは、スマホからの出庫依頼を実行する出庫JOBと自動倉庫からの入庫要求を受けて実行する入庫JOBが存在する。
- 各自動倉庫は、出庫JOBと入庫JOBは独立して実行でき、サーバーはこれを分けて配信する。
- 出庫JOBは、JOB生成に成功したあと、要求にある品種と同一商品およびそれを保管する自動倉庫をJOBに紐づけ、対象の自動倉庫へ配信する。
  - 主にスマホからの指示を直接送信するプッシュ型配信と、自動倉庫からの問い合わせに応じて指示を返送するプル型配信を行う。
  - 出庫JOBの進捗途中に異常が発生した際は、商品および自動倉庫の再割当を試行する。
- 入庫JOBは、自動倉庫から入庫要求を受けて、その要求にある品種に対して個別IDを振り、JOBを作成して自動倉庫に続行させる。
- JOBは進捗状態をもち、これを管理する。
  - 一部の進捗状態には時限を設定し、規定時間内に次の状態へ遷移しているかを監視する。
  - 規定時間を超えた、時限付き状態はタイムアウトとして処理される。

## JOBの作成の流れ

1. そのJOBは作成可能か？
    - 出庫：品種バリデーション、在庫確認
    - 入庫：品種バリデーション、空き容量確認
2. JOB番号を採番する
3. 商品ID、自動倉庫は割り当てが可能か？
    - 出庫：作成時点では未割当とする
    - 入庫：[商品ID採番ロジック](商品ID採番ロジック)を参照
4. JOBのデータをDBに登録する
5. JOBの配信および通知を行う
    - 出庫：自動倉庫へプッシュ、スマホにレスポンスを返す
    - 入庫：自動倉庫へ入庫JOBを返却

### 詳細な流れ

　スマホが複数台あるとき、出庫依頼が競合したとき、採番時の重複が発生を回避する

1. トランザクションを開始
2. そのJOBが作成可能か？
3. 当日の作成済みJOBをロック付きで検索
4. 最大連番を取得する
5. 最大連番＋1でJOB番号を作る
6. JOBに必要な情報、自動倉庫の割当情報などを準備する
7. JOBをINSERTする
8. コミット

### 作成時の競合

コミット前に競合した場合
- 後続のJOB作成処理は、当日のJOB一覧の読み取り時にロック解除待機状態となる
- ロック解除後、先行処理で作成されたJOBを含めた状態で最大連番を取得できる
- 取得した最大連番をもとに、次のJOB番号を採番して処理を継続する

コミット後に競合した場合
- 後続のJOB作成処理は、更新後のJOB一覧を通常通り取得する
- 取得した最大連番をもとに、次のJOB番号を採番して処理を継続する

補足
- JOB番号はPKなので、JOB番号が重複するとINSERTに失敗する
  - 上記の競合回避策により通常発生しない事象である
  - したがって、INSERT失敗した場合は、処理実装の不具合として、例外エラーとする

## 商品ID採番ロジック

- 入庫JOBでは、商品の割当や自動倉庫の割当が不要だが、品種コードから商品IDの採番を行う。
- 複数の自動倉庫から入庫要求が来た場合、採番ロジックが競合する可能性がある。
- 最新連番の取得->連番更新を1トランザクションで行い、処理が競合しないようにする。

### 手順

1. （作成ロジックからトランザクション継続中）
2. 品種コードに対応する品種マスタ行を、ロック付きで取得する（UPDLOCK）
3. 採番日時と最新連番を取得する
4. 採番日時が今日の日付と異なる場合は1、同じ場合は最新連番+1を次連番とする
5. 品種マスタの最新連番を次連番、採番日時を現在日時に更新する
6. 採番した商品IDを作成ロジックへ返す

### 競合時の挙動

コミット前に競合した場合
- 後から来た入庫要求側が、データ読みとり時点でロック解除待機状態となる
- ロック解除後、更新後の最新連番・採番日時を取得し、処理を継続する。

コミット後に競合した場合
- 後から来た入庫要求側が、更新後の最新連番・採番日時を通常通り取得し、処理を継続する。

## 出庫JOBの配信

出庫JOBの配信は、以下の2方式が存在する

### プッシュ型配信（サーバー -> 自動倉庫）

　スマホ操作やタイムアウトを起点とするイベントで、自動倉庫への指示を行う目的で利用する。

- スマホからの出庫依頼により、出庫JOBを作成したとき
- JOBが異常により中断した後、再割当が実行されたとき

### プル型配信（自動倉庫 -> サーバー -> 自動倉庫）

　自動倉庫からのアクションを起点とするイベントで、自動倉庫への指示を行う目的で利用する。

- 未割当JOBをキューイングした状態で、自動倉庫の定時間JOB要求を受信したとき
- 未割当JOBをキューイングした状態で、自動倉庫からのオンライン要求を受信したとき
- 自動倉庫からの入庫依頼を受信し、入庫JOBを作成したとき

## 出庫JOB配信の手順

1. トランザクションを開始
2. 配信相手の自動倉庫が、JOBを持っていないか確認する
3. 未割当JOBのうち、最も登録の古いJOBを取得する
4. JOBをAssignedに更新する
5. 商品をReservedに更新する
6. 自動倉庫に実行中JOB番号を設定する
7. コミット

### 出庫JOB配信の競合

- プッシュ型配信とプル型配信では、起点タイミングが異なるので競合する可能性がある。
- JOBデータの読み取り->状態更新を1トランザクションで行い、処理が競合しないようにする。

コミット前に競合した場合
- 後から来た配信サービス側が、JOBの割当の更新時にロック解除待機となる
- ロック解除後に、UPDATEの条件から外れるため、更新レコード数が0件となる＝更新失敗　-> ロールバックを実行
- 更新失敗を受けて「JOBなし」を返す（またはプッシュを行わない）

コミット後に競合した場合
- 自動倉庫がJOBを持っていることを確認して処理を終了する　-> ロールバックを実行
- 「JOBなし」を返す（またはプッシュを行わない）

## JOB状態

|id|name|備考|
|:---|:---|:------|
|1|Unassigned|JOBが作成されたが、商品が割当られていない|
|2|Assigned|JOBに商品を割り当て、自動倉庫に指示を出した|
|3|Picking|自動倉庫が出庫を始めた|
|4|WaitOut|出庫が完了したが、商品が取り出されていない|
|5|Putaway|自動倉庫が入庫を始めた|
|6|Completed|JOBが完了した|
|7|Canceled|JOBがキャンセルされた|
|8|Recovering|商品の再割当を待っている|
|9|Aborted|再割当可能な商品が存在せず、JOBが成立しなくなった|

## JOB状態遷移

JOBの状態は、サーバー基準で遷移するものと、外部からの指示・報告を受けて遷移するものがある

- サーバー基準：Assigned、Recovering、Aborted

- 自動倉庫報告：Picking、WaitOut、Putaway、Completed

## タイムアウト

- JOBは、以下の状態のとき、時限をもつ
  - Unassigned：出庫のみ、JOBが作成されたが、商品が割り当てられていない
  - Assigned：JOBに商品が割り当てられて、自動倉庫に配信した
  - Picking：自動倉庫が出庫作業を開始した
  - WaitOut：出庫のみ、出庫動作が完了し人搬送を待っている
  - Putaway：自動倉庫が入庫作業を開始した
- サーバーは、定時間ごとに上記状態にあるJOBの経過時間を確認し、超過を確認したJOBを異常とみなす

### 設定時間

- 今回は、デモ用に短めに設定する
- appsettings.jsonを利用して設定する

|状態|タイムアウト時間(s)|
|:---|:---|
|Unassigned|30|
|Assigned|15|
|Picking|30|
|WaitOut|30|
|Putaway|30|

読み方：この状態のときに、○○秒経過したらタイムアウト扱いにする

## 異常発生時

商品の搬送中に異常が発生したときは、JOBを存続させるために、商品を再割当を行う。

### 異常の定義

- タイムアウト : 規定時間内に状態が遷移しなかった
- アラーム報告 : JOBを配信した装置からアラーム報告があった
- オンライン要求 : JOBを配信した装置からオンライン報告があった -> アラーム報告なしで再起動したとみなす

### 再割当ロジック

JOBの中断
1. （作業中の異常発生）
2. トランザクション開始
3. JOB状態ごとの商品の扱いを踏襲し、商品の在庫状態を変更する
4. 対象自動倉庫をオフラインにする
5. JOBの商品ID、装置IDをNULLに変え、JOB状態をRecoveringに変える
6. コミット

JOBの復旧
1. （上記処理のあと続いて実行）
2. トランザクション開始
3. 同一品種の在庫を、在庫情報から検索する
4. 割当可能は商品があれば
    - 商品をReservedにする
    - JOB.item_id / equipment_id を設定する
    - JOB状態を Assigned にする
5. 割当可能は商品がなければ
    - 在庫情報がない -> Aborted
    - 在庫はあるが、現在作業ができない -> Recoveringのまま
6. コミット

JOBの配信
1. AssignedのJOBをプッシュ配信する

### 異常時の商品の取り扱い

- Assigned以前の中断 → 商品を保管中に戻す
- Picking以降の中断 → 商品を在庫から除外/削除する
