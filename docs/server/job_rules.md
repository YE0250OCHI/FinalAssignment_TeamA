# JOB設計

## 概要

- JOBは、スマホからの出庫依頼、自動倉庫からの入庫要求を、アプリの処理単位としてまとめたデータのことである。
  - 出庫JOB：スマホからの出庫依頼を実行するためのJOB
  - 入庫JOB：自動倉庫からの入庫要求を受けて実行するためのJOB
- 各自動倉庫は、出庫JOBと入庫JOBは独立して実行でき、サーバーはこれを分けて配信する。
- 出庫JOBは、JOB生成に成功したあと、要求にある品種と同一商品およびそれを保管する自動倉庫をJOBに紐づけ、対象の自動倉庫へ配信する。
  - 主にスマホからの指示を直接送信するプッシュ型配信と、自動倉庫からの問い合わせに応じて指示を返送するプル型配信を行う。
  - 出庫JOBの進捗途中に異常が発生した際は、商品および自動倉庫の再割当を試行する。
- 入庫JOBは、自動倉庫から入庫要求を受けて、その要求にある品種に対して個別IDを振り、JOBを作成して自動倉庫に続行させる。
- JOBは進捗状態をもち、これを管理する。
  - 一部の進捗状態には時限を設定し、規定時間内に次の状態へ遷移しているかを監視する。
  - 時限付き状態が、規定時間を超えても状態遷移をしなかった場合はタイムアウト（異常）として処理される。

## ライフサイクル

1. スマホの出庫依頼、自動倉庫の入庫要求からJOBが作成される
2. 自動倉庫にJOBが配信される
3. 自動倉庫からJOBの進捗報告を受信し、JOB状態を遷移する
4. JOBが最終状態まで遷移したものは履歴扱いとなる
    - 正常に最後まで遷移したものは完了
    - 異常が発生したものは異常終了
    - キャンセルされたものはキャンセル済み

## JOB作成

- 出庫JOBは、スマホからの出庫依頼によって作成される
- 入庫JOBは、自動倉庫からの入庫要求によって作成される
- JOB作成は１つのトランザクション中で行い、状態の不整合がでないようにする
  - このトランザクション中に、割り当てロジックは入れない（作成のみを感心ごととする）

### JOBの採番と作成

JOB番号の採番に競合しないように、1トランザクション中に採番を行う。

1. 各種端末から、出庫依頼や入庫要求を受信する
2. トランザクションを開始
3. 品種コードの検証（UPDLOCK、HOLDLOCK）
4. そのJOBが作成可能か？
    - 出庫：在庫確認
    - 入庫：空き容量確認、入庫JOB競合
5. 入庫JOBの場合は、商品IDの採番を試みる
6. 当日の作成済みJOBをロック付きで検索
7. 最大連番を取得する
8. 最大連番＋1でJOB番号を作る
9. その他必要情報を埋め、UnassignedでJOBデータを作る
10. JOBをINSERTする
11. コミット

コミット前に競合した場合
- 後続のJOB作成処理は、当日のJOB一覧の読み取り時にロック解除待機状態となる
- ロック解除後、先行処理で作成されたJOBを含めた状態で最大連番を取得できる
- 取得した最大連番をもとに、次のJOB番号を採番して処理を継続する

コミット後に競合した場合
- 後続のJOB作成処理は、更新後のJOB一覧を通常通り取得する
- 取得した最大連番をもとに、次のJOB番号を採番して処理を継続する

補足
- JOB番号はPKなので、JOB番号が重複するとINSERTに失敗する
- ただし、上記の競合回避策により通常発生しない事象である
- したがって、INSERT失敗した場合は、処理実装の不具合として、例外エラーとする

## 商品ID採番ロジック

- 入庫JOBでは、品種コードから商品IDの採番を行う。
- 複数の自動倉庫から入庫要求が来た場合、採番ロジックが競合する可能性がある。
- 最新連番の取得->連番更新を1トランザクションで行い、処理が競合しないようにする。

1. （作成ロジックからトランザクション継続中）
2. 品種コードに対応する品種マスタ行から、採番日時と最新連番を取得する
3. 採番日時が今日の日付と異なる場合は1、同じ場合は最新連番+1を次連番とする
4. 品種マスタの最新連番を次連番、採番日時を現在日時に更新する
5. 採番した商品IDを作成ロジックへ返す

コミット前に競合した場合
- 後から来た入庫要求側が、データ読みとり時点でロック解除待機状態となる
- ロック解除後、更新後の最新連番・採番日時を取得し、処理を継続する。

コミット後に競合した場合
- 後から来た入庫要求側が、更新後の最新連番・採番日時を通常通り取得し、処理を継続する。

## 出庫JOBの配信

出庫JOBの配信は、以下の2方式が存在する

### プッシュ配信（サーバー -> 自動倉庫）

スマホ操作を起点とするイベントで、自動倉庫への直接指示を試みる目的で利用する。

- スマホからの出庫依頼を受信し、出庫JOBの作成に成功したとき

### プル配信（自動倉庫 -> サーバー -> 自動倉庫）

自動倉庫からのアクションを起点とするイベントで、自動倉庫への指示を行う目的で利用する。

- 未割当JOBが存在し、自動倉庫からのプル要求を受信したとき
- 自動倉庫からの入庫依頼を受信し、入庫JOBの作成に成功したとき

### 出庫JOB配信の手順

- プッシュ型配信とプル型配信では、起点タイミングが異なるので競合する可能性がある。
- JOBデータの読み取り->状態更新を1トランザクションで行い、処理が競合しないようにする。

1. トランザクションを開始
2. 配信相手の自動倉庫が、JOBを持っていないか確認する
3. 未割当JOBのうち、最も登録の古いJOBを取得する
4. JOBをAssignedに更新する
5. 商品をReservedに更新する
6. 自動倉庫に実行中JOB番号を設定する
7. コミット

コミット前に競合した場合
- 後から来た配信サービス側が、JOBの割当の更新時にロック解除待機となる
- ロック解除後に、UPDATEの条件から外れるため、更新レコード数が0件となる＝更新失敗　-> ロールバックを実行
- 更新失敗を受けて「JOBなし」を返す（またはプッシュを行わない）

コミット後に競合した場合
- 自動倉庫がJOBを持っていることを確認して処理を終了する　-> ロールバックを実行
- 「JOBなし」を返す（またはプッシュを行わない）

### プッシュ配信のタイムアウト

- プッシュ配信は、レスポンス応答時間に時限をつける
- 時限を超過したら、HTTP通信のタイムアウトとして処理する

- JOBに割り当てた、

## JOB状態

|id|name|備考|
|:---|:---|:------|
|1|Unassigned|JOBが作成されたが、商品が割当られていない|
|2|Assigned|JOBに商品を割り当て、自動倉庫に指示を出した|
|3|Transferring|自動倉庫が搬送作業中である|
|4|WaitOut|搬送完了したが、商品が取り出されていない（出庫のみ）|
|5|Completed|JOBが正常に完了した|
|6|Canceled|JOBがキャンセルされた|
|7|Aborted|JOの進捗を失敗した|

## JOB状態遷移

JOBの状態は、サーバー基準で遷移するものと、外部からの指示・報告を受けて遷移するものがある

- スマホ指示：Unassigned、Canceled
- サーバー内処理：Assigned、Aborted
- 自動倉庫報告：Transferring、WaitOut、Completed

## タイムアウト

- JOBは、以下の状態のときに時限をもち、経過時間が時限を超過した際に、タイムアウト扱いとする
  - Unassigned -> 出庫可能な在庫がないために割り当てができないとき
  - Assigned -> 自動倉庫からInitiateが返ってこないとき
  - Transferring -> 自動倉庫からCompleteが返ってこないとき
  - WaitOut -> 自動倉庫からRemoveが返ってこないとき
- タイムアウト扱いになった、JOBは異常終了される

### 設定時間

- 今回は、デモ用に短めに設定する
- appsettings.jsonを利用して設定する

|状態|タイムアウト時間(s)|
|:---|:---|
|Unassigned|120|
|Assigned|30|
|Transferring|30|
|WaitOut|30|

### 異常時の商品の取り扱い

Unassigned以外の状態では、商品を管理対象外とする（出庫済み扱い）
