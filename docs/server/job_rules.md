# JOB仕様

## 概要

- JOBは、スマホからの出庫依頼を実行する出庫JOBと自動倉庫からの入庫要求を受けて実行する入庫JOBが存在する。
- 出庫JOBは、JOB生成に成功したあと、要求にある品種と同一商品およびそれを保管する自動倉庫をJOBに紐づけ、対象の自動倉庫へ配信する。
  - 主にスマホからの指示を直接送信するプッシュ型配信と、自動倉庫からの問い合わせに応じて指示を返送するプル型配信を行う。
  - 出庫JOBの進捗途中に異常が発生した際は、商品および自動倉庫の再割当を試行する。
- 入庫JOBは、自動倉庫から入庫要求を受けて、その要求にある品種に対して個別IDを振り、JOBを作成して自動倉庫に続行させる。
- JOBは進捗状態をもち、これを管理する。
  - 一部の進捗状態には時限を設定し、規定時間内に次の状態へ遷移しているかを監視する。
  - 規定時間を超えた、時限付き状態はタイムアウトとして処理される。

## JOBの作成

1. スマホで品種を指定し、出庫依頼を送信する。
2. 出庫依頼に品種の商品が出庫可能であるか確認する。
    - システム全体でその品種の商品を在庫し、かつそれが他の出庫依頼で使用予定にないか？　→　在庫なければ、JOBの作成をしない
3. 対応可能な自動倉庫、商品の保管状況を精査する。
    - その商品を保管している、対応可能な自動倉庫が存在するか？　→　なければ、未割当JOBとして登録する。
4. その在庫の中で、最も登録の古い商品をJOBに割り当てる。
5. JOBを対象の自動倉庫に、プッシュで配信する。
6. スマホに作成完了を通知する。

## 配信のタイミング

### プッシュ型配信（サーバー -> 自動倉庫）

　スマホ操作やタイムアウトを起点とするイベントで、自動倉庫への指示を行う目的で利用する。

- スマホからの出庫依頼により、出庫JOBを作成したとき
- JOBが異常により中断した後、再割当が実行されたとき

### プル型配信（自動倉庫 -> サーバー -> 自動倉庫）

　自動倉庫からのアクションを起点とするイベントで、自動倉庫への指示を行う目的で利用する。

- 未割当JOBをキューイングした状態で、自動倉庫の定時間JOB要求を受信したとき
- 未割当JOBをキューイングした状態で、自動倉庫からのオンライン要求を受信したとき
- 自動倉庫からの入庫依頼を受信し、入庫JOBを作成したとき

## 出庫JOBの競合

- プッシュ型配信とプル型配信では、起点タイミングが異なるので競合する可能性がある。
- JOBデータの読み取り->状態更新を1トランザクションで行い、処理が競合しないようにする。

### 参照と更新の手順

1. トランザクションを開始
2. 配信相手の自動倉庫が、JOBを持っていないか確認する
3. 未割当JOBのうち、最も登録の古いJOBを取得する
4. JOBを割当済みに更新する
5. 商品を予約中に更新する
6. 自動倉庫に、実行中JOBを設定する
7. コミット

### 競合時の挙動

コミット前に競合した場合
- 後から来た配信サービス側が、JOBの割当の更新時にロック解除待機となる
- ロック解除後に、UPDATEの条件から外れるため、更新レコード数が0件となる＝更新失敗　-> ロールバックを実行
- 更新失敗を受けて「JOBなし」を返す（またはプッシュを行わない）

コミット後に競合した場合
- 自動倉庫がJOBを持っていることを確認して処理を終了する　-> ロールバックを実行
- 「JOBなし」を返す（またはプッシュを行わない）
