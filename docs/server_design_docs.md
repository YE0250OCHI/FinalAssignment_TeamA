# APIサーバー基本設計書

## 概要

　本書は、自動倉庫管理システムにおけるサーバー機能の外部仕様を定義するものである。   
　本システムは、スマートフォン端末からの入出庫依頼および自動倉庫設備からの作業要求・進捗報告を受け付け、JOBの管理および在庫管理を行う。サーバーはスマートフォン端末および自動倉庫設備との通信をHTTP REST APIにより行う。  
　APIの詳細仕様（URI、リクエスト・レスポンス形式、ステータスコード等）は別紙「API仕様書」に定義する。

## システム構成

　本システムは、スマートフォン端末、サーバー、自動倉庫設備で構成する。  
サーバーは、スマートフォン端末および自動倉庫設備からの要求を受け付け、JOB管理、在庫管理、棚管理を行う中心機能として動作する。

```text
スマートフォン端末
    ↓ ↑
  HTTP REST API
    ↓ ↑
サーバー
    ↓ ↑
  HTTP REST API
    ↓ ↑
自動倉庫設備
```

### スマートフォン端末

|項目|内容|
|---|---|
|接続方式|HTTP REST API|
|役割|入出庫依頼、JOB状況確認、商品受取報告|
|主な送信情報|出庫依頼、入庫依頼、商品受取報告|
|主な受信情報|JOB情報、JOB状態、エラー情報|

### 自動倉庫設備

|項目|内容|
|---|---|
|接続方式|HTTP REST API|
|役割|JOB取得、作業実行、進捗報告、エラー報告|
|主な送信情報|オンライン通知、進捗報告、完了報告、エラー情報、入庫要求|
|主な受信情報|出庫JOB、入庫JOB、棚使用状況|

## 業務フロー

### 出庫

1. スマートフォン端末が出庫依頼を登録する。
2. サーバーは出庫JOBを生成する。
3. サーバーは自動倉庫設備へ出庫JOBを配信する。
4. 自動倉庫設備は出庫作業を開始する。
5. 自動倉庫設備は作業進捗をサーバーへ報告する。
6. 自動倉庫設備は商品を取出口へ搬送する。
7. スマートフォン端末に搬送完了が通知される。
8. 作業者が自動倉庫設備から商品を取り出す。
9. 自動倉庫設備から取出し完了が報告される。
10. サーバーはJOBを完了とする。

```text
スマホ
  ↓ 出庫依頼
サーバー
  ↓ JOB生成
自動倉庫
  ↓ 作業開始
  ↓ 作業完了
商品搬送
  ↓
商品取出し
  ↓ 取出し
JOB完了
```

### 入庫

1. 自動倉庫設備が入庫対象品種を通知する。
2. サーバーは入庫JOBを生成する。
3. 自動倉庫設備は入庫作業を開始する。
4. 自動倉庫設備は商品を棚へ格納する。
5. 自動倉庫設備は作業完了を報告する。
6. サーバーは在庫情報を更新し、JOBを完了とする。

```text
スマホ
  ↓ 出庫依頼
サーバー
  ↓ JOB生成
自動倉庫
  ↓ 作業開始
  ↓ 作業完了
商品搬送
  ↓
商品取出し
  ↓ 取出し
JOB完了
```

### エラー発生時

1. エラー発生フロー
2. 自動倉庫設備でエラーが発生する。
3. 自動倉庫設備はサーバーへエラー情報を通知する。
4. サーバーはエラーログを記録する。
5. 対象自動倉庫設備を切り替えてリトライする、不可能なら対象JOBを異常終了とする。

``` text
自動倉庫
  ↓ エラー通知
サーバー
  ↓ ログ記録
  ↓　　　　　　　↓
自動倉庫切替　JOB異常終了
```

## JOB配信仕様

- 出庫JOBは、自動倉庫設備へ配信される。
- 配信方式はプッシュ配信およびプル配信の両方に対応する。

### 出庫JOB：プッシュ配信

- サーバーは、出庫JOBが生成された際、対象設備へJOBを通知する。

### 出庫JOB：プル配信

- 自動倉庫設備は、サーバーへ次の出庫JOBを要求できる。
- サーバーは、対象設備に割り当てられた未着手JOBが存在する場合、そのJOBを返却する。
- 未着手JOBが存在しない場合は、JOBなしとして応答する。

### 入庫JOB

- 入庫JOBは、自動倉庫設備からの入庫要求により生成される。
- サーバーは要求内容をもとに入庫JOBを登録し、作業対象情報を返却する。

### JOB配信順序

- 同一設備に複数の未完了JOBが存在する場合、JOBは登録順に処理する。

### 二重配信防止

- 同一JOBは、同一設備に対して重複して処理されないものとする。
- サーバーはJOB状態を管理し、既に処理開始済みのJOBを再配信しない。
- JOBに割当済みの商品は、別のJOBに割当ができないようにする。

### オフライン時の扱い

- 設備がオフライン状態の場合、サーバーは新規JOBを配信しない。
- 設備がオンライン復帰した後、当該設備に割り当て済みの未完了JOBについて配信を再開する。
- オンライン復帰時の配信は、作業中断前の状態を復旧することを目的とした再配信とする。
- オンライン復帰を契機として、未割当JOBの新規割当は行わない。

## エラー方針・制約

- サーバーは、要求内容に問題がある場合、処理を実行せずエラーとして応答する。
- サーバー内部で異常が発生した場合、処理を中断しエラーとして応答する。
- エラー発生時は、原因調査のためログを出力する。
- 業務エラー発生時は、対象データを変更しない。
- 通信エラー発生時は、要求を受け付けない。
- システムエラー発生時は処理を中断し、エラーログを出力する。

### 業務エラーの定義

- 存在しないJOBが指定された場合
- 存在しない商品が指定された場合
- 在庫が不足している場合
- 棚容量を超過する場合
- 現在状態では実行できない操作が要求された場合

### 通信エラーの定義

- 未登録端末からの要求
- リクエスト形式不正
- 必須項目不足

### 通信エラーの定義

- データベース接続異常
- 想定外例外の発生
- システム資源不足

### エラーログの出力

エラー発生時は以下の情報をログへ記録する。

- 発生日時
- 要求元設備
- 対象JOB
- エラー内容
