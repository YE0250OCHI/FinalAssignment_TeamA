# コンソール設計資料

部品を保管する自動保管棚を模擬したコンソールアプリ。  
サーバーからの出庫指示を受け取って出庫動作を行うほか、コンソール入力によって入庫動作を行う。  
在庫情報はサーバー側のDBに保存するため、入庫時は登録情報をサーバーに送信する。

## 状態

- オフライン
- 待機中
- 作業中

### オフライン

- 起動直後および異常発生時に移行
- サーバーへオンライン化報告を実行
- 在庫情報（商品番号：品種ID-入庫日-枝番）と空き容量を送信

### 待機状態

- 入庫操作待機 (コンソール入力)
- サーバーからの出庫指示 (POST) を待機
- 5秒毎にサーバーへ出庫指示確認 (GET送信)
  - 入庫・出庫は同時に動作可能
  - 出庫のPOSTとGETはどちらかが動作中はもう片方は無効

### 入庫動作の流れ

- コンソール入力を待機 (品種ID)
- サーバーに入庫情報 (品種ID) を送信
- サーバーからのレスポンス（JOB番号・商品番号）を受信
- 正常レスポンス
  - 入庫開始フラグを送信
  - 入庫動作の完了を待機 (今回はコンソール入力で代用)
  - 入庫完了フラグを送信
  - 待機モードに戻る
- エラーレスポンス（422）
  - エラー内容を表示
  - 待機状態に戻る 
- ESCで待機状態に戻る

### 出庫動作の流れ

- 出庫要求を受信
- 出庫開始フラグを送信
- コンソールに出庫指示の内容を表示
- 出庫動作の完了を待機 (今回はコンソール入力で代用)
- 出庫完了フラグを送信
- ワーク取出の完了を待機 (今回はコンソール入力で代用)
- 取出完了フラグを送信
- 待機状態に戻る

## DB定義

棚マスタ（テーブル名：master）
|No.|項目名|カラム名|データ型|長さ|NOT NULL|PK|備考|
|---:|:---|:---|:---|:---:|:---:|:---:|:---|
|1.|棚番号|block_id|int||〇|〇|1～50|
|2.|ブロック名|block_name|varchar|20|〇||A-1～A-50|
|2.|使用可能フラグ|block_status|bit||〇||Default:1|

棚在庫テーブル（テーブル名：inventory）
|No.|項目名|カラム名|データ型|長さ|NOT NULL|PK|備考|
|---:|:---|:---|:---|:---:|:---:|:---:|:---|
|1.|棚番号|block_id|int||〇|〇|1～50|
|2.|商品個別ID|item_id|varchar|50|||品種ID-入庫日-枝番|

## 表示

アプリ起動
``` text
========== 入出庫棚 ==========
オンライン通知：サーバーからのレスポンスを待っています……


```

出庫動作
``` text
============ 出庫 ============
F1:出庫操作モード

==============================
JOB番号：XXXX
商品番号：XXX-YYYYMMDD-ZZZZ
出庫完了(Enter)＞
取出完了(Enter)＞

```

入庫動作
``` text
============ 入庫 ============
F2:入庫操作モード
ESC:キャンセル
==============================
品種ID＞XXX
JOB番号：XXXX
入庫完了(Enter)＞
```

異常表示
``` text
異常が発生しました
REBOOT：Enter END：ESC
＞
```

## ログ

Nlog
- システムログ
- 入出庫ログ

### システムログ

- 通信の内容やエラーを記録する

### 入出庫ログ

- 入庫・出庫の内容を記録する
- 動作完了または異常終了でログ更新
- 動作|方向|商品番号

## 備考

- 収納数50個
- ~~アプリ起動時にサーバーに対して出庫指示がないかGETで確認~~
- 入出庫作業中にサーバーからPOSTが来た場合”422：遷移拒否”レスポンスを返す
- コンソール操作で非常停止のエラーが送れるようにする
  - 発生時はオフライン状態に移行し、コンソール操作でオンライン化からやり直す
  - 実行中JOBは異常終了したものとして、ワークを除去する
- 在庫情報の更新タイミング
  - 出庫：作業開始時点で取り出されたものとして在庫情報は削除する
  - 入庫：作業完了まで在庫情報は登録しない
- 通信のイレギュラーは基本異常終了
- Ctrl＋Cへの対策を追加
